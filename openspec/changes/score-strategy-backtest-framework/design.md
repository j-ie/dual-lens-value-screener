## Context

当前项目已经具备投资质量分数计算能力，但缺少面向历史时点的统一回测执行能力。策略评估主要依赖局部样例和主观判断，难以回答“分数是否具备稳定预测力”“策略参数是否过拟合”等关键问题。

回测能力需要跨越多个模块：历史特征构建、历史行情收益计算、组合构建、调仓执行、绩效评估与结果持久化。该变更属于跨层改造，且涉及时间一致性与结果可复现等关键约束，适合先明确技术设计再实施。

## Goals / Non-Goals

**Goals:**
- 提供可重复执行的分数策略回测流程，确保时点一致、无未来函数。
- 支持两类评估模式：分数分层回测与规则选股组合回测。
- 提供统一绩效指标与结果落库能力，支持后续前端/接口查询与对比。
- 保持与现有投资质量评分逻辑解耦，允许后续替换或扩展评分源。

**Non-Goals:**
- 不在首版引入复杂组合优化（如风险预算、凸优化求解）。
- 不在首版实现分钟级/日内高频回测，仅覆盖日频收益与低频调仓。
- 不在首版实现全量可视化分析平台，仅提供任务与结果数据接口。

## Decisions

1. 采用“回测任务”聚合根作为领域入口  
   - 方案：定义 BacktestJob（参数、状态、时间范围、版本信息）与 BacktestResult（指标、曲线摘要、分层统计）作为核心实体。  
   - 原因：统一任务生命周期，便于幂等重跑、失败恢复与审计。  
   - 备选：直接脚本化回测并仅导出文件；缺点是难以追踪参数与结果对应关系。

2. 采用“时点快照构建器”作为特征来源统一门面  
   - 方案：新增历史特征构建组件，按调仓日 T 严格读取 T 前可见财务与行情快照，统一输出评分输入结构。  
   - 原因：把“时点一致”从策略逻辑中抽离，避免各策略重复处理公告滞后细节。  
   - 备选：在策略内部自行拼接历史字段；缺点是容易引入未来函数且难以测试。

3. 回测引擎分层为“信号生成 + 组合构建 + 执行记账 + 指标评估”  
   - 方案：  
     - SignalStage：生成分数、分组或规则入选结果；  
     - PortfolioStage：按规则构造目标权重；  
     - ExecutionStage：按调仓频率更新持仓并计入交易成本；  
     - MetricsStage：计算收益、回撤、换手、IC 等指标。  
   - 原因：职责清晰，便于单元测试与后续策略扩展。  
   - 备选：单函数串行处理；缺点是耦合高、调试困难。

4. 首版采用等权持仓 + 可配置过滤约束  
   - 方案：支持 Top-N / Top-Quantile 等权持仓；过滤条件包括停牌、ST、最小成交额（若可得）、行业可选约束。  
   - 原因：先保证可解释性和可复现性，降低参数维度。  
   - 备选：引入市值加权或风险平价；首版复杂度较高且调参空间过大。

5. 结果存储采用“指标宽表 + 曲线点表（或压缩 JSON）”  
   - 方案：任务表存参数与状态，结果表存核心指标，曲线明细按日期存净值/基准/超额。  
   - 原因：兼顾查询效率与可追溯性，支持后续前端绘图。  
   - 备选：仅存本地文件；缺点是服务化查询能力不足。

## Risks / Trade-offs

- [历史财报可得性口径错误] → 使用公告日与披露滞后规则建模，并在回测日志写入每期样本构建时间戳。
- [幸存者偏差导致收益虚高] → 股票池按历史时点构建，覆盖退市与停牌状态，不使用“当前成分股回看”。
- [交易约束简化导致偏乐观] → 首版强制计入成本参数并在报告中显式标注未覆盖的约束项。
- [计算耗时较长] → 调仓粒度首版限定为周/月/季；提供按时间区间分批执行能力。
- [策略过拟合] → 强制输出样本外区间指标，后续补充 walk-forward 评估任务。

## Migration Plan

1. 新增回测领域模型、应用服务与基础仓储，确保不影响现有筛选与详情流程。  
2. 增量创建回测任务与结果数据表，采用向后兼容迁移脚本。  
3. 在 CLI/API 增加回测触发入口与结果查询接口。  
4. 灰度运行历史区间回测，与离线基准结果比对后再开放给前端页面。  
5. 若出现严重偏差，可关闭回测入口并回滚到迁移前版本（不影响已有核心筛选功能）。

## Open Questions

- 财报公告滞后规则是统一固定天数，还是按报表类型（年报/季报）分层配置？
- 首版基准指数默认使用哪一条（如沪深300/中证全指），是否允许任务级覆盖？
- 曲线明细是否全部入库，还是按采样/压缩策略保存以控制存储成本？
