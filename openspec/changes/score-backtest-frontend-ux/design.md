## Context

分数策略回测能力正在建设中，但当前系统主要围绕批跑与详情展示，尚无专门的回测操作入口。若前端不同时设计，用户将只能通过接口或脚本发起任务，无法高效完成参数迭代、结果比较与失败排查。

本变更需要同时覆盖前后端：后端提供稳定、可前端消费的任务与结果接口；前端提供“创建任务 → 跟踪执行 → 查看结果 → 复制参数重跑”的闭环交互。

## Goals / Non-Goals

**Goals:**
- 提供前端可操作的回测工作台，降低回测试验门槛。
- 提供结构化参数表单与输入校验，减少非法配置导致的任务失败。
- 提供任务列表与详情页，支持状态追踪、错误反馈、参数回显与复跑。
- 提供结果可视化，展示净值曲线、核心指标、分层收益与 IC 诊断。
- 后端接口输出统一结构，便于前端稳定渲染与后续扩展。

**Non-Goals:**
- 首版不实现复杂图表编辑器或拖拽式策略搭建。
- 首版不做多任务并发调度控制台，仅展示任务状态与结果。
- 首版不做全量权限系统改造，沿用现有服务访问模型。

## Decisions

1. 前端采用“工作台 + 详情页”双层结构  
   - 方案：新增回测入口页（任务列表 + 新建抽屉/弹窗）与任务详情页（结果分析）。  
   - 原因：列表页适合管理多次实验，详情页适合深度分析与复盘。  
   - 备选：单页承载所有内容；缺点是信息密度过高，交互复杂。

2. 参数模型采用“后端优先定义、前端按契约渲染”  
   - 方案：后端提供稳定参数字段与默认值，前端按字段分组渲染（基础参数/选股参数/交易成本/过滤条件）。  
   - 原因：降低前后端口径偏差，减少参数兼容问题。  
   - 备选：前端自定义参数模型；缺点是后端升级时容易断裂。

3. 任务状态采用轮询机制，前端显示阶段进度  
   - 方案：任务列表与详情按固定间隔轮询状态；显示 pending/running/success/failed 与阶段信息。  
   - 原因：实现成本低、与现有接口风格一致。  
   - 备选：WebSocket 推送；缺点是改造成本和运维复杂度更高。

4. 结果展示先覆盖“研究刚需指标”  
   - 方案：优先支持净值与基准曲线、收益/回撤/夏普、换手、IC/RankIC、分层收益摘要。  
   - 原因：首版满足策略有效性判断与参数对比核心需求。  
   - 备选：一次性上线全量高级分析模块；缺点是交付周期长。

5. 提供“参数复制重跑”而非“在线编辑历史任务”  
   - 方案：历史任务只读，支持一键复制参数到新建任务。  
   - 原因：保证任务审计一致性，避免历史结果被误解为可变。  
   - 备选：允许编辑后重跑原任务；缺点是审计语义不清晰。

## Risks / Trade-offs

- [前后端参数字段不一致] → 以后端 schema 为唯一来源，新增字段需同步更新前端映射与校验。
- [任务轮询频率过高导致压力上升] → 列表页与详情页采用分级轮询间隔，成功/失败后停止轮询。
- [图表复杂度高导致首版体验不稳定] → 首版先提供关键图表与表格，保留扩展位。
- [失败原因不透明导致用户误操作] → 任务详情明确展示错误摘要与建议动作（修参重跑/稍后重试）。

## Migration Plan

1. 后端先补齐回测任务与结果查询接口，确保字段结构稳定。  
2. 前端新增回测工作台页面并接入路由入口。  
3. 前端完成新建任务、任务列表、详情分析与重跑交互。  
4. 联调验证参数校验、状态轮询、结果渲染与异常提示。  
5. 灰度发布并收集反馈，逐步扩展高级分析能力。
