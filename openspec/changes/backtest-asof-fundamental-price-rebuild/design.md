## Context

当前回测依赖 `financial_snapshot.fetched_at` 构造时点，属于“任务沉淀型快照回测”。该模式在数据刚起步时只能覆盖很短区间，不适合策略研究与参数稳定性评估。项目内已存在较完整的三大财报历史表和数据同步能力，但尚未把这些历史数据系统性用于回测重建。

需要将回测切换为“交易日驱动 + as-of 重建”模式：每个调仓日先根据公告可见性重建财务特征，再结合历史行情计算未来收益，最终形成可跨多年复现的回测结果。

## Goals / Non-Goals

**Goals:**
- 以历史交易日为主轴，支持多年回测而非依赖近期快照沉淀。
- 实现财报 as-of 重建，确保每个调仓日只使用当时可见数据。
- 引入历史行情驱动收益计算，提升收益与基准比较的可信度。
- 输出更强诊断信息（覆盖区间、样本可用率、剔除原因）。
- 前端/接口返回数据可用区间提示，降低用户配置错误率。

**Non-Goals:**
- 首版不实现分钟级/日内回测。
- 首版不实现复杂风险模型或组合优化器。
- 首版不强依赖新增外部商业数据源，优先复用已有 TuShare 链路。

## Decisions

1. 回测主轴改为“交易日历 + 再平衡频率”  
   - 方案：从历史行情表读取交易日序列，按周/月/季频率抽取调仓点。  
   - 原因：交易日粒度天然适配收益计算与基准比较。  
   - 备选：继续按快照 `fetched_at` 驱动；缺点是区间短且不可控。

2. 特征重建采用“as-of 快照拼装器”  
   - 方案：在调仓日 T，按公告日期 `ann_date <= T` 取各报表最近可见记录，拼成评分输入。  
   - 原因：避免未来函数，保证研究结论可复现。  
   - 备选：按报告期截止日 `end_date` 直接取最近值；缺点是可能穿越。

3. 收益计算采用“持仓区间行情收益 + 成本扣减”  
   - 方案：用调仓日到下一调仓日的历史行情计算标的收益，组合收益按等权聚合并扣交易成本。  
   - 原因：收益定义清晰，与现有回测指标兼容。  
   - 备选：继续使用市值快照差值代理；缺点是噪声大且缺口径一致性。

4. 数据覆盖诊断作为一等输出  
   - 方案：落库并返回：可用日期区间、有效样本数、剔除原因计数、无效期占比。  
   - 原因：帮助用户快速判断“是策略问题还是数据问题”。  
   - 备选：仅返回失败文案；缺点是定位效率低。

5. 前端增加“可用区间约束提示”  
   - 方案：创建任务前拉取可用日期范围；若用户区间越界，前端阻止提交并给出引导。  
   - 原因：把错误前移，减少无效任务。  
   - 备选：仅后端报错；缺点是交互成本高。

## Risks / Trade-offs

- [公告日期字段缺失或不一致] → 建立兜底规则（ann_date 缺失时回退到 end_date + 滞后天数），并在诊断中标注。
- [历史行情覆盖不完整] → 在回测前做覆盖校验，不足时提示并允许降级到较短区间。
- [重建过程计算开销上升] → 增加按日期批处理与中间缓存，避免重复拼装。
- [旧快照逻辑切换导致行为变化] → 保留配置开关，允许短期并行比对新旧口径。

## Migration Plan

1. 新增历史行情读取与交易日历访问层。  
2. 新增财报 as-of 重建组件并接入回测引擎。  
3. 切换收益计算到历史行情路径，保留旧逻辑开关用于比对。  
4. 增加诊断字段与接口返回结构。  
5. 前端接入可用区间提示与提交前校验。  
6. 用固定样例区间回归并对比旧版结果后切换默认路径。
